# Investigation: Embedded checkleft checks and their replacement path

Status: investigation / recommendation. No code changes. This document enumerates every check that is _compiled into the checkleft binary_ today, and for each one recommends a replacement bucket as it (potentially) moves out of the binary toward the external check tiers.

## What "embedded" means here

checkleft has two disjoint families of checks:

- **Embedded (built-in) checks** — compiled into the `checkleft` binary. They are registered by `register_builtin_checks` in `src/checks/mod.rs`, each implements the in-process `Check` / `ConfiguredCheck` trait pair, and each receives a `&ChangeSet` + `&dyn SourceTree` and runs native Rust logic (regex, tree-sitter, YAML parsing) directly against the change-scoped file model.
- **External checks** — resolved from packages, _not_ baked into the binary. They come in two tiers (`src/external/mod.rs`):
  - **`component-v1`** ("component mode"): a Rust crate compiled to a `wasm32-wasip2` WebAssembly Component Model artifact, run under `wasmtime` in a capability-scoped WASI sandbox. This tier is **implemented and in production** — the first-party wasm checks under `checks/` (`file/size`, `file/forbidden-path`, `file/ifchange`, `rust/giant-structs`, `md/link-integrity`) are compiled into the multiplexed `//tools/checkleft/preinstalled-bundle` component and shipped.
  - **`declarative-v1`** ("declarative mode"): a _zero-code_ YAML manifest that tells the framework how to find, invoke, and reshape the output of an _existing external binary_ (linter/formatter). This tier is also implemented — the `checks/lint/*.yaml` and `checks/format/*.yaml` checks wrap rustfmt, clippy, Biome, prettier, oxc, etc. with no Rust code.

"Embedded check" in this document therefore means specifically **the 13 built-in checks in `register_builtin_checks`** — the ones we want to evaluate for a move out of the binary.

## Two framing corrections (read before the recommendations)

The task brief references an in-flight "language-agnostic structured-change core (P2070)" and a "Starlark-backed checks direction." Grounding in the repo turned up two things worth stating plainly, because they change what the recommendations can depend on:

1. **There is no `P2070` / "structured-change core" artifact in the repo.** Nothing under `tools/checkleft` (or the wider workspace, excluding build outputs) mentions `P2070`, `structured-change`, or "language-agnostic structured." The _closest real concept_ is the **structured `ChangeSet` / hunk-level diff model** that already exists: it is defined in `src/input.rs` and mirrored across the WIT boundary as `change-set` / `file-diff` / `diff-hunk` records in `wit/check.wit` (project P1446, the wasm component-model work). This is a _diff_-structured model, **not** a language AST. Its companion is the change-detection work in `docs/designs/robust-change-detection-in-checkleft.md` (project P844), which produces the changeset CI-agnostically. **Consequence:** the recommendations below do **not** block on a not-yet-built structured-change core. The structured changeset already exists and both external tiers already consume it. Where I say a migration "depends on" something, it is a concrete, identifiable dependency (an access scope, a rule kind, a transform primitive), not "P2070 landing."

2. **"Starlark-backed checks" is a misread — checks are not authored in Starlark.** `bazel-policies` is a _compiled Rust built-in that parses Bazel Starlark source_ (BUILD/MODULE.bazel/.bzl) via `tree-sitter-starlark` (`src/checks/bazel/starlark.rs`). The design docs (`docs/designs/bazel-policy-checks.md`, `docs/designs/code-patterns.md`) **explicitly reject** exposing Starlark — or a generic tree-sitter query DSL — as the check _authoring/config_ language ("One Generic Starlark Query DSL — Rejected"). The lever for "declarative, not compiled" is a **typed rule vocabulary in YAML config** (e.g. `kind: forbidden_rule_call`) consumed by a compiled engine, plus the two external tiers. So "Starlark" is the _subject language being parsed_, not the way checks are written. This matters below because it means the `*-policies` checks stay compiled by design; only their _rules_ are declarative.

A third clarification that sharpens the buckets: **`declarative-v1` mode only wraps external binaries.** Its transforms reshape a tool's stdout/exit code (`json`/`linelist`/`passthrough`); there is no "run this regex over file content" declarative primitive (the `regex` and `sarif` transforms are reserved-but-unimplemented, per `docs/spikes/buildifier-declarative/SPIKE-NOTES.md`). So a check whose logic is native regex/AST matching **cannot** become a `declarative-v1` manifest — its only "no new code" homes are (a) staying a typed-config built-in, or (b) becoming config of a general **wasm** engine. I use "declarative-config" below to mean "expressible as typed rules with no new code," and call out explicitly when `declarative-v1` external mode is _not_ an option.

## Cross-cutting findings

### Cluster A — the Bazel-policy family (`bazel-policies`, `bazelrc-policies`, `bazelversion-policies`, `repo-visibility`)

These four are one conceptual surface: typed policy rules over Bazel-ecosystem files. `repo-visibility` is **already redundant** — its behavior is a single `forbidden_package_default_visibility` rule on `bazel-policies` that exists in code today. The two engines (`bazel-policies` Starlark parser, `bazelrc-policies` rc parser) and the trivial `bazelversion-policies` should **stay compiled**; only their rule sets are declarative config. Net: delete `repo-visibility`, keep three compiled engines fed by `CHECKS.yaml` rules.

### Cluster B — the content-pattern family (`forbidden-imports-deps`, `frontend-no-legacy-api`, `typo`, `todo-expiry`)

All four are "regex over changed-file lines, scoped by include/exclude globs, with severity + remediation." They differ only in specialization: `frontend-no-legacy-api` = import-line regex; `typo` = word-boundary terms + suggested replacement; `todo-expiry` = a _required_ (not forbidden) pattern. **Recommendation: define one general content-pattern check** supporting both forbid-match and require-match rules, glob scoping, and optional suggested-fix, then express the other three as config (bucket 3) and `todo-expiry` as a require-rule (bucket 1).

Key constraint: because `declarative-v1` cannot run regex over file content, this general check must be **either a kept built-in or a `component-v1` wasm check** — it is _not_ a `declarative-v1` manifest. Given the wasm tier is already in production, a single `content-patterns` wasm check is an attractive home and would remove four built-ins for the price of one extension. `forbidden-imports-deps` is the natural anchor to generalize from.

### Cluster C — the workflow-policy family (`workflow-action-version`, `workflow-run-patterns`, `workflow-shell-strict`)

All three parse GitHub Actions YAML and apply a typed policy (pinned action versions, forbidden run-script regex, required strict shell). They mirror the Bazel-policy pattern but for a different file type. **Recommendation: one `workflow-policies` `component-v1` check** with typed rules, replacing three built-ins. Pure changeset computation, no host internals → clean wasm fit.

### Checks with no clean external replacement (load-bearing)

- **`bazel-policies` and `code-patterns` engines** — general, AST-driven, deliberately compiled per their design docs. They are migration _targets_, not sources. A wasm port is theoretically possible (tree-sitter compiles to wasm) but carries no forcing need and Large effort; leave compiled unless there is a concrete reason to extract them.
- **Framework plumbing is not a check and cannot be a bucket** — change detection / VCS diffing, file selection + exclusion enforcement, the sandbox builder, `needs` binary resolution, and the declarative transform engine itself are what _make_ the external tiers possible. They stay in the binary by necessity.

### Dependencies and ordering

1. **`repo-visibility` → `bazel-policies` config** (bucket 3). Zero new dependencies — the rule kind exists. Delete the check, migrate config. Lowest risk; do first as the proof-of-pattern.
2. **`frontend-no-legacy-api` → `forbidden-imports-deps` config** (bucket 3). Zero new dependencies.
3. **General content-pattern check** (cluster B). Requires a design decision (kept built-in vs. new `component-v1`) and a rule model that covers forbid + require + word/suggested-fix. Then fold `typo`, `todo-expiry`, `frontend-no-legacy-api` in. Not blocked by any unbuilt infrastructure.
4. **`workflow-policies` wasm check** (cluster C). Depends only on the already-shipping `component-v1` tier + a YAML parser in wasm. Fold the three workflow checks in.
5. **`rust-test-rule-coverage` → wasm** (bucket 2). Depends on the `declared-files` access scope (already implemented and exercised by `file/ifchange`). Revisit the brittle `rust_test(` substring match during the port.
6. **`bazel-policies` / `code-patterns` engines** — no action; keep compiled. Reconsider a wasm port only if there is appetite to remove tree-sitter from the binary.

Suggested sequencing rationale: steps 1–2 are near-free deletions that validate the "delete bespoke, express as config of a general check" motion. Steps 3–4 each retire three-to-four built-ins behind one general check and are independent of one another (do in either order). Step 5 is a self-contained wasm port. Step 6 is explicitly deferred.

## Summary of buckets

- **Bucket 1 (declarative-config, no new code):** `todo-expiry`, `forbidden-imports-deps` (anchor), `bazelversion-policies`, `bazelrc-policies`.
- **Bucket 2 (wasm `component-v1`):** `workflow-action-version`, `workflow-run-patterns`, `workflow-shell-strict`, `rust-test-rule-coverage`.
- **Bucket 3 (usage of an existing general check):** `repo-visibility` (→ `bazel-policies`), `frontend-no-legacy-api` (→ `forbidden-imports-deps`), `typo` (→ general content-pattern check).
- **Flagged / keep compiled (load-bearing engines):** `bazel-policies`, `code-patterns`.

## Follow-up code changes (out of scope for this investigation)

These are the concrete code changes implied by the recommendations, listed for the operator to file separately (this PR is docs-only):

- Delete `repo-visibility` and migrate its usage to a `bazel-policies` `forbidden_package_default_visibility` rule.
- Retire `frontend-no-legacy-api` in favor of `forbidden-imports-deps` config.
- Design + build a general `content-patterns` check (forbid + require + word/suggested-fix rules), then retire `typo` and `todo-expiry`.
- Build a `workflow-policies` `component-v1` wasm check and retire the three `workflow-*` built-ins.
- Port `rust-test-rule-coverage` to `component-v1` with the `declared-files` access scope; revisit the `rust_test` detection heuristic.
