## TL;DR

I enumerated every user-facing surface the design specifies and checked each against shipped code on `main`. **No incident-002-class silent regression exists** — no surface has been deleted, 404'd, or downgraded to a hardcoded placeholder/stub. Every surface the design ships is live and, for the badge/chip surfaces, backed by tests.

**19 surfaces checked: 17 PRESENT, 2 with deviations worth an operator's attention:**

- **macOS "Engine" attempts view** (§DQ11) — the attempt-list surface exists and is fully functional, but shipped as an **"Engine" filter segment inside the Activity Log window** rather than the **top-level "Engine" tab** the design named, and the attempt **kind** is surfaced in the free-text "Headline"/"Source" columns rather than a dedicated **`Kind` column**. This is a UI-placement deviation, **not** a behavioral downgrade or a placeholder — all attempt data (conflict + CI) is listed, filterable, and drillable. Classified **PRESENT (deviation)**.
- **macOS per-chore CI-remediation detail view** (§DQ11) — present in the Activity Log detail pane with failed checks, triage class, log excerpt, and timestamps, but **missing the "per-file fix summary" and a computed "elapsed time"** the design's sentence lists, and it is attempt-scoped rather than chore-scoped. Classified **PARTIAL**.

Neither deviation is the failure mode this sweep exists to catch (a merged surface silently deleted during a forward-port). Both are live, discoverable, and tested. They are polish/completeness gaps the operator may optionally file as follow-ups (§Follow-ups), not restorations of a lost surface.

## Method and a note on the "§Surfacing" heading

The task brief points at a `§Surfacing` heading. **The design doc has no section literally titled "Surfacing."** Its concrete user-facing surfaces are enumerated across three sections instead:

- **§Design Question 2 — Parent State Model** → the kanban blocked badges (`### Kanban presentation`, `### Composed kanban presentation`).
- **§Design Question 8 — Coordination with `review-feedback`** → the `### FrontendEvent shape` variants.
- **§Design Question 11 — CLI Surface** → the `boss engine …` verbs and the `### macOS app surface` (tab, PR-card chips, detail view).

Plus the backing protocol/schema surfaces from §DQ2/§DQ3 and the Implementation Phases table (§Implementation Phases, items #12–#15, #34–#37). I swept all of them.

**Important context — the Post-Unification Reconciliation.** The design carries a `## Post-Unification Reconciliation — Revision-Based Parent-State Model` section (added when `unify-pr-remediation-on-revisions`, PR #971, moved the fix vehicle from a bespoke ephemeral execution to a real `revision` task). Under the reconciled model, **while a conflict/CI-fix revision is in flight the parent stays in `in_review`** (the revision card in Doing is the "something is happening" signal); the parent only flips to `blocked: merge_conflict` on terminal cases (churn cap exhausted, `create_revision` refused, revision terminated). This means the `blocked: merge conflict` kanban badge (surface #1) now appears **less often** by design — this is an intentional evolution, not a regression, and the badge-rendering code is still present.

Evidence gathered by three parallel read-only code sweeps (CLI/`cli`, engine/`engine`+`protocol`, macOS/`app-macos`) plus direct spot-reads. All paths relative to `tools/boss/`.

## Detail on the two non-PRESENT surfaces

### #12 — macOS "Engine" attempts view: PRESENT but relocated, no `Kind` column

**Design (§DQ11, L1669–1671):** _"The macOS app gets a new 'Engine' tab … showing both `rebase_attempts` and `conflict_resolutions` in a single table view, with a `Kind` column."_ Extended for CI at L1725–1727 (_"The Engine tab grows a `ci` row kind"_).

**Shipped:** The attempt-list capability is fully present but lives as an **"Engine" segment of the `ActivitySourceFilter`** inside the Activity Log window (`app-macos/Sources/ActivityLogView.swift:10`), which merges `conflict_resolutions` + `ci_remediations` into `EngineAttemptRow`s (`:176–192`) and renders them in a sortable table with a detail pane. It is **not** a top-level `NavigationMode` tab — those are only Agents / Work / Designs / Automations (`app-macos/Sources/Models.swift:3–9`, iterated at `app-macos/Sources/ContentView.swift:214–221`). The table columns are **Time / Source / Headline / Outcome / Work Item / ID** (`:377–419`); there is **no dedicated `Kind` column** — attempt kind is folded into the free-text Headline (`"\(kindLabel): …"`, `:66`) and the "Source" column shows `dispatch`/`engine`.

**Why this is PRESENT (deviation), not a silent regression:** the surface is live, discoverable (Cmd-Shift-L → Engine filter), lists both attempt kinds, and drills into full per-attempt detail. Nothing was deleted or stubbed. Two named affordances from the design's sentence — the _top-level tab placement_ and the _dedicated `Kind` column_ — are not literally present; the kind information is available in a different column. `rebase_attempts` (a sibling `auto-rebase-stacked-prs` surface, not owned by this project) is not merged into this view, but that is out of scope for this design's surfaces. Code comments still call it "the Engine-tab" (`app-macos/Sources/ChatViewModel.swift:279–287`), confirming this is the intended surface, relocated.

### #15 — macOS per-chore CI detail view: PARTIAL

**Design (§DQ11, L1733):** _"A detail view per chore shows the failed checks, the worker's triage decision, the log excerpt, the per-file fix summary, the elapsed time."_

**Shipped:** `ActivityEngineDetailPane` (`app-macos/Sources/ActivityLogView.swift:663–770`) surfaces failed checks, triage class, log excerpt, head SHA, attempt kind, consumes-budget, and Created/Finished timestamps. **Missing:** the "per-file fix summary" and a computed "elapsed time" (only raw Created/Finished strings, not a duration). It is also attempt-scoped in the Activity Log pane rather than a dedicated per-chore panel. No evidence these two fields ever shipped — so this is a completeness gap vs the design spec, not a regression from a prior state.

## Conclusion

**No surface specified by the design is ABSENT, deleted, or downgraded to a placeholder/stub.** The incident-002 failure mode (a merged surface silently vanishing during a forward-port) did **not** recur here — every enumerated surface resolves to live, generally test-backed shipped code. The two deviations (#12 relocated Engine view / no `Kind` column; #15 CI detail missing per-file fix summary + elapsed time) are polish/completeness gaps, not lost surfaces, and are surfaced below as optional follow-ups for the operator to file separately.
