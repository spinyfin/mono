## Problem

Boss's merge button currently executes one hardcoded verb: `gh pr merge --auto --squash`. Once flunge flips to queue-only enforcement (main pushes restricted to the Trunk GitHub app), that verb hard-fails there. This design makes the merge mechanism a per-product choice — `direct` (today's behavior, which also transparently covers mono's GitHub-native merge queue) vs `trunk_queue` (submit to Trunk's queue via its REST API and track the entry asynchronously to a terminal state) — and surfaces queue progress in the existing Merging UI lane, with queue eviction flowing into the existing CI-failure remediation machinery.

## Goals

- **Per-product merge mechanism.** A product-level setting selects how an approved merge is executed: `direct` (today's `gh pr merge --auto --squash`) or `trunk_queue` (Trunk REST API submit). Flunge is the first `trunk_queue` adopter; mono is unchanged.
- **Asynchronous queue-backed merge verb.** For `trunk_queue` products, the merge button submits the PR to Trunk's queue and the engine tracks the entry through `pending → testing → merged | failed | cancelled`. Merge-approval semantics upstream of the click are untouched — only the mechanics of executing an approved merge change.
- **Queue state in the Merging UI.** The existing Merging lane shows Trunk-queued PRs with their queue state and position, on freshness good enough to watch a merge land (≤ ~30 s staleness while anything is enqueued).
- **Eviction is a first-class failure signal.** A PR kicked from the queue (combined CI failed on its `trunk-merge/*` test branch) flows into the existing ci_watch → engine-triggered-revision remediation path, exactly like a red PR-branch CI run or a GitHub merge-queue rebounce, and is resubmitted once fixed.
- **Loud, honest failure semantics.** Missing/expired token, Trunk API unreachable, queue paused — every failure mode surfaces as an explicit error or attention item. The engine never silently falls back to `gh pr merge` for a `trunk_queue` product.
- **A safe enforcement flip.** A documented sequence and runbook for making flunge queue-only, including how the engine detects and reports the "enforcement is on but Boss is misconfigured" state.

## Non-goals

- **Changing merge-approval semantics.** Who may click merge, review gating (`review_required_state`), and the `InReview` lifecycle before the click are all unchanged. This design only changes what happens after the click.
- **Webhook ingestion in v1.** The engine runs on the operator's Mac with no public ingress; Trunk's Svix-delivered webhooks cannot reach it. Polling is the v1 mechanism; a webhook relay is sketched as future work only.
- **Migrating mono to Trunk.** Mono keeps GitHub's native merge queue via the existing `gh pr merge --auto` verb; no mono behavior changes in this project. "Non-goal" means out of this project's scope, not architecturally ruled out: flunge is deliberately a **trial** — if the queue proves better, mono can follow, and the design keeps that a config flip (`boss product set-merge-mechanism mono trunk_queue` plus the branch-protection change) rather than a code change. Nothing in the merge path, poller, UI, or remediation hardcodes flunge: `merge_mechanism` is a per-product column with no hardcoded product list, `TrunkQueueProbe` keys on `(repo, target_branch)`, repo coordinates derive from `repo_remote_url`, and eviction rides the shared remediation substrate. The prerequisites for a later mono switch are: multi-org token support (task 14) if mono lands in a separate Trunk org (until then the single-token blast radius in the risks section covers both repos), 429/`Retry-After` handling at mono's request volume (built into `trunk_client` from day one), and gating the GitHub-native `mergeQueueEntry`/rebounce observation on `merge_mechanism != trunk_queue` so a repo can never be observed in both queues at once (built in v1 — see the coordination rules).
- **Queue administration from Boss.** Creating/deleting/pausing queues, changing concurrency/batching modes, and `setImpactedTargets` optimization stay in the Trunk web app. Boss reads queue state; the only write verbs it ever issues are entry-level — `submitPullRequest` and `cancelPullRequest`. `restartTestsOnPullRequest` was scoped as a flake-retrigger verb but is **not** wired in v1 (see the eviction section) — never queue-lifecycle administration.
- **Driving Buildkite.** Trunk triggers CI on `trunk-merge/*` branches and Buildkite already builds all branches (validated in the flunge trial). Boss does not orchestrate CI for queue runs; it only reads failure evidence when an eviction happens.
- **ETA prediction in v1.** `getMergeQueueMetrics` returns Prometheus text suitable for dashboards; deriving a per-PR ETA from it is speculative and deferred (`future / not a v1 blocker` in the task list).
- **Priority UI in v1.** The API accepts a `priority` on submit and the client plumbs it (`SubmitPullRequestRequest.priority`), but the merge button submits at default priority (the field is omitted). A priority picker is future work.

## Trunk API surface (verified 2026-07-19 against docs.trunk.io / openapi.json)

- Base URL `https://api.trunk.io/v1`, auth header `x-api-token: <org token>`, JSON bodies, all endpoints POST except metrics.
- `POST /v1/submitPullRequest` — `{repo:{host,owner,name}, pr:{number}, targetBranch, priority?, noBatch?}` → `200 {}`. `priority` is int-or-string, optional.
- `POST /v1/getSubmittedPullRequest` — `{repo, pr, targetBranch}` → `{id, state, readiness, stateChangedAt, priorityValue, priorityName, prNumber, prTitle, prSha, prBaseBranch, prAuthor}`.
- `POST /v1/listPullRequests` — `{repo, targetBranch, state?, since?, cursor?, take? (1-100, default 50)}` → `{pullRequests[], nextCursor?}`. `since` filters concluded PRs by timestamp — the reconciliation backstop.
- `POST /v1/getQueue` — `{repo, targetBranch}` → `{state, branch, concurrency, testingTimeoutMinutes, mode, batch, enqueuedPullRequests[]}`. One call returns every enqueued PR — the cheap bulk probe.
- `POST /v1/cancelPullRequest`, `POST /v1/restartTestsOnPullRequest` — dequeue / re-test verbs. (`cancelPullRequest` is wired in v1; `restartTestsOnPullRequest` is modeled in the client but deliberately not wired into the poller.)
- `GET /v1/getMergeQueueMetrics` — Prometheus text (dashboards; not machine-friendly per-PR ETA).
- **PR states:** `not_ready`, `pending`, `testing`, `tests_passed`, `merged`, `failed`, `cancelled`, `pending_failure`.
- **Queue states:** `RUNNING`, `PAUSED`, `DRAINING`, `SWITCHING_MODES`.
- No documented rate limits; bodies capped at 20 MiB. Webhooks exist (Svix-delivered, `pull_request.*` and `pull_request_batch.*` families) but require a public HTTPS endpoint.
- Non-API submit paths (`/trunk merge` comment, enqueue label, Chrome extension, `trunk merge` CLI) exist and were used in the flunge trial; humans may keep using them alongside Boss.

## Alternatives considered

### A. Keep the comment command: engine posts `/trunk merge` on the PR

The trial mechanism. Rejected as the engine's verb: the comment is fire-and-forget with no structured acknowledgement (a typo'd or unauthorized comment fails silently), attribution is "whoever's GitHub token posted a comment" rather than an org API token, there is no priority control, and the engine would still need the REST API anyway to read queue state. The REST `submitPullRequest` is attributable, synchronous-acknowledged, and priority-capable. (Humans commenting `/trunk merge` by hand remains supported — the poller treats externally-submitted entries as observed state, see coordination rules.)

### B. Model Trunk as a third arm of the existing GitHub merge-queue observation (no mechanism setting)

Teach the merge poller to always probe Trunk for every PR and treat "a Trunk queue exists for this repo" as the routing signal, avoiding a product-level setting. Rejected: it makes merge routing an emergent property of remote state rather than declared intent — a Trunk org token outage would silently degrade routing decisions, and pre-enforcement (when both `gh pr merge` and the queue work) the choice would be ambiguous. A declared per-product mechanism is inspectable, testable, and fails loudly when its prerequisites are missing.

### C. Synchronous merge verb: submit and block until terminal state

Keep the merge action synchronous by having the handler wait for the queue outcome. Rejected outright: queue runs take as long as a full CI cycle (tens of minutes on flunge), batching/bisection can multiply that, and the engine's RPC handlers must not hold work-item state hostage to a remote CI pipeline. The whole point of the queue is that "merge" becomes a tracked asynchronous process; Boss already has exactly the right shape for this (fire the verb, let the poller observe the outcome) in the GitHub-native path.

### D. A third mechanism value for mono's GitHub-native queue

Considered making the setting three-valued: `direct` | `github_queue` | `trunk_queue`. Rejected: `gh pr merge --auto --squash` is literally the same verb whether the repo has no queue (merges/arms auto-merge) or a GitHub-native queue (enqueues) — GitHub decides, and the poller already observes `mergeQueueEntry` either way. A `github_queue` value would add config surface with zero behavioral difference and create a misconfiguration class (mono set to `direct` vs `github_queue` behaving identically but reading differently). Mono stays under `direct` semantics; the docs for the setting say "direct = GitHub-native merge verb, including GitHub's own merge queue where enabled."

## Chosen approach

### Per-product merge mechanism

New nullable TEXT column `products.merge_mechanism`, values `'direct'` (NULL ⇒ `direct`) and `'trunk_queue'`, added via a guarded `ALTER TABLE` (`migrate_products_merge_mechanism`, `migrations_b.rs`), surfaced on the protocol `Product` **and** `CreateProductInput` structs (`#[serde(default, skip_serializing_if = "Option::is_none")]`, builder-pattern field per repo convention) and settable via `boss product set-merge-mechanism <selector> [--mechanism <direct|trunk_queue> | --unset]` plus the app's product settings. `map_product` coerces an empty string to `None`. Parsed engine-side (`engine/core/src/merge_mechanism.rs`) into:

```rust
pub enum MergeMechanism {
    Direct,               // gh pr merge --auto --squash (covers GH-native queue)
    TrunkQueue { target_branch: String /* default "main" */ },
}
```

`MergeMechanism::parse(Option<&str>)` maps `None`/`Some("direct")` → `Direct`, `Some("trunk_queue")` → `TrunkQueue { "main" }`, and **any other value** to a loud `bail!` error (the CLI passes the raw string through; validation is server-side, not at the clap layer). The repo coordinates Trunk uses (`host`, `owner`, `name`) derive from the product's existing `repo_remote_url` (flunge → `github.com` / `brianduff` / `flunge`) — no new columns. The target branch defaults to `main`; the `TrunkQueue.target_branch` field can currently only ever hold `"main"` — a `products.trunk_target_branch` override column is deliberately deferred until a product needs it.

The wire verb is `FrontendRequest::SetProductMergeMechanism { product_id, mechanism: Option<String> }` (handler `products::handle_set_product_merge_mechanism`, DB method `WorkDb::set_product_merge_mechanism`, event `product_merge_mechanism_set`). Both create and set paths validate through `MergeMechanism::parse` before writing.

There is no per-task override. Merge mechanism is a property of how a repo integrates with its trunk, not of individual work items.

### `trunk_client` crate

Per the crates-over-modules convention (precedent: the `claude_client` extraction, PR #1702), the REST client shipped as a new crate `tools/boss/trunk_client` (crate name `boss_trunk_client`, edition 2024, bazel visibility limited to `//tools/boss/engine/core:__pkg__`):

- Typed request/response structs for `submitPullRequest`, `getSubmittedPullRequest`, `listPullRequests`, `getQueue`, `cancelPullRequest`, `restartTestsOnPullRequest`; a `TrunkPrState` enum for the eight PR states and `TrunkQueueState` for the four queue states, both with an `Unknown(String)` variant and `#[serde(from = "String", into = "String")]` round-tripping so new Trunk states degrade gracefully. Supporting types: `TrunkRepoRef{host,owner,name}`, `TrunkPrRef{number}`, `TrunkPrLookup{repo,pr,target_branch}` (shared by getSubmitted/cancel/restart), `TrunkPullRequest` (carries `id` and `state_changed_at` — the eviction dedup key), `TrunkQueue`, `ListPullRequestsResponse`, `TrunkPriority`.
- Takes the token as a constructor parameter (`Arc<dyn TrunkTokenProvider>`, plus `StaticTokenProvider`); the crate knows nothing about keychains or engine config, and `SecretString` redacts in `Debug`. Edges stay one-directional: `engine → trunk_client`, never the reverse.
- **Divergence — shared `http_retry` crate.** The plan scoped retry/backoff to `trunk_client`; as built, `RetryPolicy`, jittered backoff (±25 %), and the process-wide `reqwest::Client` singleton were extracted into a new shared crate `tools/boss/http_retry`, and `claude_client` was refactored onto it (which changed `claude_client::RetryPolicy::new` from a 2-arg to a 3-arg constructor and touched `planner.rs`). `trunk_client` defaults: 4 attempts / 250 ms base / 30 s cap, 15 s per-attempt timeout. Retries on 5xx, transport errors, **and 429**, honoring `Retry-After` (integer-seconds only; HTTP-date form is a documented deferral) clamped to `max_backoff`; no retry on any other 4xx.
- **Divergence — five error variants, not four.** Structured errors are `TrunkError::{Auth, NotFound, QueueUnavailable, Transport, Decode}` — `Decode` was split from `Transport` so a schema mismatch is never retried. 401/403 surfaces as `Auth` so callers can emit the "token missing/expired" attention; 429-after-exhaustion collapses into `QueueUnavailable`.
- The crate landed with no production consumer (task 1 is standalone scaffolding); the engine wiring arrived in tasks 3–7.

### Auth: the Trunk org API token

- **Provisioned** by the operator from the Trunk web app (org-level API token) and handed to Boss via `boss engine trunk set-token` (reads from stdin when piped, else a non-echoing `rpassword` prompt — never argv) or the app's Settings → Trunk pane. `boss engine trunk status` reports token presence + source and runs a live `getQueue` smoke check against the first `trunk_queue` product.
- **Stored** in the macOS Keychain: service `dev.spinyfin.boss.trunk`, account `api-token@trunk.io`, accessibility `AfterFirstUnlockThisDeviceOnly`. Env override `BOSS_TRUNK_API_TOKEN` for development. **Never** in any repo, never in the DB, never logged.
- **As built — two extracted crates.** Rather than mirror the GitHub `KeychainTokenStore`, the implementation extracted a shared `boss-keychain` crate (`tools/boss/keychain`; macOS Data Protection Keychain in release, `0600` file fallback in dev, `keyring` elsewhere, plus the `SecTask` entitlement-probe FFI) and re-based **both** the Trunk and pre-existing GitHub stores on it. The Trunk store lives in a new `boss-trunk-auth` crate (`tools/boss/trunk_auth`), which implements both `TrunkTokenSource` (for the set/status handlers) and `boss_trunk_client::TrunkTokenProvider` (for live API calls).
- **RPC surface.** `FrontendRequest::{TrunkSetToken { token }, TrunkStatus}`; reply `FrontendEvent::TrunkStatus { configured, source, queue_check: Option<TrunkQueueCheckDto{ok, detail}>, note }`. Handlers `handle_trunk_set_token` / `handle_trunk_status` (`engine/core/src/app/trunk_auth.rs`). The token store is a trait object on `ServerState` for test injection. The engine trims the token and rejects empty/whitespace before storing.
- **Smoke-check timing note.** The live `getQueue` smoke check was initially deferred at token-provisioning time (task 3) because `products.merge_mechanism` had not yet landed in that workspace — `trunk status` returned `queue_check: None` with a note. It was wired up later (async `trunk_queue_smoke_check`, run against the first `trunk_queue`-mechanism product) as a prerequisite of the pre-enforcement validation work.
- **Failure semantics (loud, no fallback):**
  - Merge click on a `trunk_queue` product with no token → the handler replies a loud `WorkError` (prefixed `merge_when_ready:`) and no fallback to `gh pr merge`.
  - Token rejected (401/403, `TrunkError::Auth`) at submit or poll time → same loud path, plus a deduplicated `trunk_token_rejected` attention item, since post-enforcement a dead token means nothing can merge.
  - Trunk unreachable → submit fails loudly with retry advice; the poller backs off exponentially (base 30 s, cap 5 min) and emits a `trunk_queue_unreachable` attention item if unreachable for > 15 min while entries are being tracked.

### The merge verb: submit + standing merge intent

`handle_merge_when_ready` keeps all existing guards, resolves the task's product, `MergeMechanism::parse`s it, and branches. `Direct` runs today's path (now via the `DirectMergeExecutor` trait). `TrunkQueue`:

1. **Record a standing merge intent** — a side-table row (`trunk_merge_intents`, migration `migrate_trunk_merge_intents_table`, schema version 27). The intent is the poller's tracking anchor and the memory that the operator approved this merge — it is what authorizes automatic resubmission after an eviction is fixed (see below). A partial unique index (`ON (work_item_id) WHERE status = 'active'`) enforces one active intent per work item; a second merge click on an already-active intent whose PR matches is a no-op that re-reports current queue state (the test asserts `submitPullRequest` is called exactly once), while a mismatched/missing intent replies `WorkError`.
2. **Submit**: `TrunkClient::submit_pull_request` with the PR number and target branch at default priority. On success, optimistically set `merge_queue_state = "queued"` and `merge_queue_detail = {source:"trunk", state:"pending"}` so the card moves to Merging immediately, publish a `trunk_merge_submitted` `work_item_changed`, and reply `MergeWhenReadyAccepted` with the new `action` value `trunk_enqueued` (`MergeAction::TrunkEnqueued.as_str()`; extends the existing wire vocabulary, which the app already switches on).
3. On failure, the just-inserted intent row is deleted (rollback), the card stays in Review, and the error is loud per the auth section — **never** a fallback to `gh pr merge`.

As-built intent columns: `id TEXT PRIMARY KEY, work_item_id, pr_url, pr_number, repo, target_branch, status, last_trunk_state, last_trunk_state_at, submit_count (DEFAULT 1), created_at`. `status` is free TEXT with values `active | merged | cancelled | exhausted`; `insert` writes `active`, and retiring to terminal values is owned by the poller (task 5) and eviction/resubmit (tasks 6–7). The submit path deliberately does **not** fire `PrReconcilerKick` (the Trunk poller's own idle rescan notices the new intent); this keeps the GitHub probe's API quota out of the Trunk cadence.

The task itself stays `InReview` throughout, exactly like the GitHub-native queue path — Merging is a presentation of `merge_queue_state`, not a `TaskStatus`. This keeps every downstream consumer (dependency cascade, design-doc population, revision gating on "chain root PR is Open") working unmodified.

**Entry state machine** (engine-side view of one intent):

```
                    submit
  [active] ──────────────────────▶ pending / not_ready ──▶ testing ──▶ tests_passed
                                        │                    │              │
                                        │                    ▼              ▼
                                        │                  failed ◀── pending_failure
                                        │                    │
                                        ▼                    ▼
                                    cancelled          eviction → ci_watch remediation
                                        │                    │ (revision reaches `done`)
                                        ▼                    ▼
                                 intent cancelled      auto-resubmit (budget-capped)
                                                       via boss:awaiting_resubmit sentinel

  terminal: trunk state = merged  → observed by existing merge poller (PR merged on GitHub) → task done
            intent cancelled      → card back to Review + attention item
            budget exhausted      → intent retired 'exhausted' + parent blocked: ci_failure_exhausted
```

Trunk's `merged` state is deliberately **not** the engine's merge-completion trigger: the existing GitHub-side probe already detects the PR's merged state and runs the whole `mark_merged()` cascade. Trunk polling only feeds the intermediate states and the failure signal, then retires the intent when it observes `merged`. This means a human who enqueues via `/trunk merge` without touching Boss still gets correct terminal behavior today — Trunk state tracking is additive.

### Queue state ingestion: polling

**Where:** in the existing merge-poller task (`spawn_loop`), as a `TrunkQueueProbe` (`tools/boss/engine/core/src/trunk_queue_poller.rs`) with its own arm in the poller's `select!`, sharing the task's lifetime and publisher plumbing. **Divergence:** unlike the plan, it does **not** inherit the 60 s sweep or the GitHub `PollTier` — it runs an independent cadence via a purpose-built `TrunkPollTier { Testing, Pending }` so the Trunk cadence never drags the GitHub probe and its API quota along. The `trunk_client` crate does the transport (behind a `TrunkQueueApi` trait for test seams).

**What:** per `(repo, target_branch)` with ≥ 1 active intent, one `getQueue` call per elapsed cycle — it returns state + every enqueued PR in one response, covering position and per-PR state for all tracked entries at once. For any active intent whose PR is absent from `enqueuedPullRequests` (it left the queue), one `getSubmittedPullRequest` call (`resolve_missing_entry`) resolves its terminal state (`merged` / `failed` / `cancelled`). A low-frequency (~10 min per queue) `listPullRequests since=<cursor>` reconciliation backstop (delivered in #2194) walks every page via `next_cursor`, runs unconditionally before `getQueue` (so an unreachable queue still gets reconciled), and advances its `since` cursor only after a full window succeeds.

**Cadence and freshness:** `Testing` tier = 15 s (any observed entry `testing`/`tests_passed`), `Pending` tier = 30 s otherwise; no Trunk API calls at all when nothing is enqueued (the loop still wakes every 15 s for a local-DB rescan to notice a new merge click, since submit does not kick the reconciler). That is a few requests/min against an API with no documented rate limit, giving the Merging UI ≤ 30 s staleness. All failures back off exponentially (base 30 s, cap 5 min) without affecting the GitHub probe.

**What it writes:** the same two columns the UI already consumes — `merge_queue_state = "queued"` (any live Trunk state) and a `merge_queue_detail` JSON — plus `trunk_merge_intents.last_trunk_state`. **As-built the detail carries eight keys**, two more than the original plan: `{source: "trunk", state, position (1-based index into enqueuedPullRequests), enqueued_at, queue_state, section_order, repo, target_branch}`. `repo`/`target_branch` were added (#2194) so the renumber pass can group entries by their Trunk queue identity — a product can hold intents against multiple queues, each with its own 1-based `position`. `enqueued_at` is derived from the intent's own `created_at` (Trunk PRs carry no enqueue timestamp). `section_order` is decoupled from `position`: it is assigned by the renumber pass below and seeded from the prior pass, while `position` remains the raw queue index the `#n` badge renders. `set_task_merge_queue_state` returns a bool (did the stored pair actually change, via SQL `IS NOT`) so the push-only app only receives a `work_item_changed` on a real change. Queue-level anomalies (`PAUSED`/`DRAINING` while intents are active) emit one deduplicated `trunk_queue_not_running` attention per non-RUNNING episode, cleared on return to RUNNING.

**Renumbering (as built):** rather than literally extend `renumber_merge_queue`, a generic lower-level helper `merge_queue_renumber::renumber_section_order<T, K>` was extracted (`merge_queue_renumber.rs`) and **both** the GitHub `renumber_merge_queue` and a new Trunk `renumber_trunk_merge_queue` were built on it. The Trunk caller runs once per product at the end of every pass, sorting by `trunk_queue_sort_key` (`(repo, target_branch)` identity first, then ascending `position`, missing values last, `task_id` tie-break) and rewriting only rows whose `section_order` changed. Without a populated `section_order` the Merging lane would tie every Trunk card at `.max` and render unordered.

The poller registers six counters (`trunk_queue_poller.{queue_probes, queue_probe_failures, entry_lookups, state_writes, intents_retired, attentions_filed}`), later joined by `evictions_detected`, `resubmits`, and `queue_cancellations`.

**Webhooks (future, explicitly not v1):** if freshness ever needs to be sub-poll-interval, a small relay (Cloudflare Worker/Tailscale Funnel endpoint receiving Svix deliveries and exposing a pull queue the engine drains) can feed the same reconciliation entry points. The poller remains the source of truth even then — webhooks would only be a cache-invalidation hint, so nothing about this design's state model changes.

### Eviction: a first-class failure signal

Trunk state `failed` (reached directly or via `pending_failure`) means the PR was kicked from the queue because combined CI failed on its `trunk-merge/*` test branch. This is the moral twin of the existing GitHub merge-queue `FAILED_CHECKS` rebounce, and as built the two paths are unified: `ci_watch::on_trunk_queue_eviction_detected` and `on_merge_queue_rebounce_detected` are thin wrappers over a shared `on_queue_side_failure_detected` carrying a `QueueSideFailureEpisode { discriminator, before_commit_sha, failure_kind, labels }`.

1. The poller's `apply_resolved_state`, on `Failed | PendingFailure` for an active intent, computes an episode discriminator, checks `ci_remediation_exists_for_head_sha_at_trigger` for idempotency, and if new calls `handle_trunk_queue_eviction` → `on_trunk_queue_eviction_detected`.
2. A `ci_remediations` row is inserted with `failure_kind = 'trunk_queue_eviction'`. **As-built idempotency:** the `(entry_id, stateChangedAt)` pair is overloaded into the existing `head_sha_at_trigger` column as the string `trunk:<id>@<stateChangedAt>` (fallback `trunk:<id>@unknown`) — **no new column, no schema change**, mirroring how rebounce overloads the same column. `before_commit_sha` is NULL for evictions. `prSha` is deliberately **not** persisted or keyed — the investigation (#2153/#2184) verified it is re-read live and mutates mid-episode.
3. **Failure evidence (as built, diverged).** The plan promised a stored Buildkite log excerpt keyed by `prSha`/queue-branch head. The investigation found that route unreliable (`prSha` is the PR head = the construction commit's _second parent_, and GitHub check-runs return zero for construction commits), so the implementation instead does an **org-wide Buildkite build search** filtered on the branch prefix `trunk-merge/pr-<N>/` (`ci_log_reader::find_trunk_merge_eviction_build`, `bk api /builds?state[]=failed&state[]=failing`, up to `MAX_EVICTION_BUILD_SEARCH_PAGES = 3`, excluding non-gating `trunk-temp/*` builds). It stores only the resulting build **coordinates** as `failed_checks` (`RequiredCheckFailure` with `target_url = <web_url>#<job-uuid>`); it does **not** pre-fetch a log excerpt. The worker-facing revision directive (`runner.rs`) tells the worker to discover the Buildkite build and fetch logs itself rather than trusting `gh pr checks`.
4. The parent flips `in_review → blocked: ci_failure` and the standard engine-triggered **revision** is spawned via the shared substrate (`consumes_budget: 1`, `attempt_kind: "fix"`), consuming one slot of the existing `ci_attempt_budget` (default 3). No new budget knob. The intent stays `active` with `last_trunk_state = failed`.
5. **Auto-resubmit (as built, diverged trigger).** When the spawned revision's task reaches `TaskStatus::Done`, `ci_watch::on_ci_resolved` marks the intent `boss:awaiting_resubmit` (a Boss-namespaced sentinel stored in `last_trunk_state`), and the poller's resubmit arm calls `submitPullRequest` again, increments `submit_count`, clears the sentinel, re-places the card in Merging, and publishes `trunk_merge_resubmitted`. **The trigger is the revision task reaching `done`, not a head-branch-CI-green probe** — head-branch CI is meaningless for a queue-side failure and would flip-flop. Auto-resubmit is bounded by the same budget; on exhaustion the intent is retired `exhausted`, its Merging columns cleared, the parent sits `blocked: ci_failure_exhausted`, and the existing attention + `boss engine ci retry` reset path applies.
6. `cancelled` (a human ran `/trunk cancel`, the queue was drained, or Boss's own `cancelPullRequest` ran) retires the intent **without** remediation: card back to Review, `trunk_queue_entry_cancelled` attention item titled "PR was removed from the Trunk merge queue", no revision spawned. Cancellation is a human decision, not a failure.

**Suppression of double-fire (as built):** rather than a failure-kind-specific "suppress `on_ci_failure_detected` for the same head SHA," an eviction is prevented from ever reading as a PR-head failure — `merge_poller::classify_ci` unconditionally drops any required check whose name starts with `"Trunk Merge Queue"` (the trunk-io app's own bookkeeping check) — plus the pre-existing "active remediation + revision in flight" guard. The queue-side vs head-SHA guards were generalized behind `is_queue_side_failure_kind(kind)` = `matches!(merge_queue_rebounce | trunk_queue_eviction)`, applied across `on_ci_in_flight_supersedes_failure`, `on_ci_resolved`, the control-verb rejection paths, and completion's cleared-signal retirement.

**Flake handling (deferred).** The design intended flake-flavored evictions to route to a budget-free retrigger via `classify_pre_triage()` (resubmit, or `restartTestsOnPullRequest` on a still-live entry). This is **not implemented in v1** — it is blocked on a mapping from Buildkite job states (`broken`/`timed_out`/`canceled`) onto the `STARTUP_FAILURE`/`CANCELLED` vocabulary `classify_pre_triage` keys on, and `restartTestsOnPullRequest` was left out of the poller's `TrunkQueueApi` trait. Tracked as follow-up work.

### Coordination with conflict_watch / ci_watch (don't misread queue states)

Rules preventing the queue's own churn from being misread as PR pathology:

- **While an intent is live** (`pending`/`testing`/`tests_passed`), the PR's head branch is untouched — Trunk tests a `trunk-merge/*` construction branch. The existing probes keep watching the head branch and can legitimately fire (e.g. main moved and the PR went `CONFLICTING`). A conflict detected mid-queue is real — Trunk will also fail it — so conflict_watch proceeds normally, but as built `conflict_watch::on_conflict_detected` marks the intent with the sentinel `boss:superseded_by_conflict` (in `last_trunk_state`, only where conflict_watch actually takes the slot — after the auto-rebase-active/foreign-bucket early returns). The poller then, before its queue-membership lookup, cancels the entry via `cancelPullRequest` and takes **no** eviction remediation (the conflict resolver owns the slot). After the conflict revision resolves, `conflict_watch::on_resolved` advances the sentinel to `boss:awaiting_resubmit` and the same auto-resubmit path runs. **Divergence note:** `superseded_by_conflict` shipped as a `last_trunk_state` sentinel string, not a persisted queue-exit status field. `mark_trunk_intent_awaiting_resubmit(allowed_from)` gates each owner to its own sub-state, so an eviction and a conflict cannot steal each other's resubmit.
- **Never double-fire.** A single queue failure produces exactly one remediation: the eviction path keys its `ci_remediations` row on `trunk:<id>@<stateChangedAt>` (idempotent across sweeps), and the `classify_ci` exclusion above keeps the post-eviction head-branch CI redness from registering as a second failure.
- **`merge_mechanism` gates observation, not just submission.** For `trunk_queue` products the GitHub-side probe stops writing `merge_queue_state`/`merge_queue_detail` and skips `check_merge_queue_rebounce()`. As built this is one helper — `is_trunk_queue_product(work_db, product_id)` = `matches!(MergeMechanism::parse(...), Ok(TrunkQueue { .. }))`, fail-open false (#2192) — feeding two enforcement points: the column-write preservation in `update_pr_poll_state` (originally landed as `preserve_merge_queue_state` in #2185) and a candidate filter in the extracted `run_merge_queue_rebounce_pass` (called from both `run_one_pass` and `reconcile_one`). This rule is required for v1 correctness: the GitHub probe recomputes the queue columns from `mergeQueueEntry`/`autoMergeRequest` every sweep (both absent for a Trunk-queued PR), so an ungated sweep would clear the Trunk-written state within one cycle. CI/review-state stamping and the merged-terminal detection that drives `mark_merged()` are unchanged — which is what keeps Trunk's `merged` handling GitHub-side.
- **`not_ready` is not a failure.** It means GitHub-side readiness (approvals, non-queue required checks) is missing. The card shows a "waiting on readiness" badge. The design also called for an attention nudge if `not_ready` persists > 30 min; **that nudge is not implemented in v1** and is tracked as follow-up work.
- **The existing auto-merge/red-CI override** (`merge_queue_state` forced to `None` when `ci_required_state == "fail"` under `auto_merge_enabled`, task.rs:549) applies only to the GitHub `auto_merge_enabled` state and is untouched; Trunk entries are governed by the intent state machine above.

### Merging UI

The app already has a Merging lane: the collapsible `done-merging` section rendered above the recency buckets in the Done column, populated by `WorkTask.boardColumn`'s `in_review where isInMergingSection → .done` routing and sorted by the engine-computed `section_order`. Trunk entries reuse the whole chain — `merge_queue_state` → `boardColumn` → `mergingSection()` → `MergeQueueBadge` — with **no changes to the routing/placement code**, exactly as designed. The Swift chrome (`ContentView.swift`, `ChatViewModel+WorkBoardSections.swift`, `ChatViewModel+EventHandling.swift`) shipped as:

- **Card chrome.** `MergeQueueDetail.parse` learns `source` and `queue_state`. For Trunk rows, computed `trunkBadgeText`: `pending` → `"#<position>"` (or `"Queued"` before a position exists), `testing` → `"Testing"`, `tests_passed` → `"Merging…"`, `not_ready` → `"Waiting on readiness"`, unknown → the raw string. `isTrunkTerminal` (`failed`/`pending_failure`/`cancelled`/`merged`) renders `EmptyView()` — those cards have already left Merging. Readiness glyph: `tests_passed` → green check, `failed`/`pending_failure` → unmergeable, everything else → clock.
- **Queue banner.** `queueStateBanner` maps `PAUSED`/`DRAINING`/`SWITCHING_MODES` (and a lowercased fallback) to a header banner shown via a new `banner:` param on `CollapsibleWorkBoardSection`, computed as the first non-RUNNING state in lane order.
- **Action feedback.** The previously-discarded `action` on `MergeWhenReadyAccepted` now drives a single-slot, auto-dismissing `MergeFeedbackNotice` (`WorkMergeFeedbackBanner`): `trunk_enqueued` → "Submitted to Trunk merge queue", plus distinct copy for the other actions. Known limitation: if the Merging section is collapsed, the feedback banner auto-dismisses unseen (accepted; section defaults expanded).
- **Failure surfacing** follows the existing bifurcation: mechanical CI-shaped failures (eviction) go through the block-reason + on-card badge channel — the card snaps back to Review with the existing `blocked: ci_failure` treatment — while needs-a-human conditions (token dead, queue paused, cancelled entry) go to attention items in `AttentionsView`. No new surfaces were invented.
- **The merge button** is unchanged in the app: same Review-card gate, same confirmation dialog, same RPC; the engine routes by product.
- **Settings UI (added scope, #2253).** A dedicated Trunk Settings pane (`TrunkSettingsPane.swift`, a "Trunk" tab in `SettingsView`) provides token entry (`SecureField`, never persisted client-side) + live status, and the product-edit sheet gained a segmented `merge_mechanism` picker (Direct / Trunk merge queue) with an inline warning + `SettingsLink` when `trunk_queue` is selected but no token is configured. This landed as **app-only** work — it emits the already-existing `set_product_merge_mechanism`, `trunk_set_token`, and `trunk_status` verbs with **no new engine/protocol/wire surface**. (This delivered UI that tasks 2/3 had committed to but that had been built locally and never pushed.)
- **Not in scope:** promoting Merging from a Done-column section to a top-level `WorkBoardColumnKey` column (future work). The UI stays a thin client throughout: it renders state the engine reconciles; no Trunk API calls from the app.

## Risks / open questions

- **Auto-resubmit authorization.** The design treats the original merge click as standing authorization to resubmit after each eviction fix (budget-capped). As built, the resubmit fires when the fix revision reaches `done`. If the operator instead wants a fresh human click per resubmission, the intent machinery still works — the resubmit step just becomes an attention item + Review-lane affordance.
- **Eviction dedup key — resolved (#2184).** The `(id, stateChangedAt)` composite key was empirically verified load-bearing: `getSubmittedPullRequest.id` is per-PR stable but **not** per-episode, so `stateChangedAt` is required to distinguish queue episodes; `prSha` is re-read live mid-episode and is unsafe for dedup/provenance. The `(pr, prSha, stateChangedAt)` fallback was rejected. The original deliberate-eviction runbook was found flawed (a scratch PR that fails its own CI never gets admitted to the queue) and rewritten: the PR must pass its own CI but fail against `main` on the construction branch.
- **`tests_passed` → `merged` gap.** Between Trunk merging the construction branch and GitHub reporting the PR merged, the card says "merging" while GitHub still says open. The existing Hot-tier GitHub probe closes this within ~15 s.
- **Buildkite log access — resolved (#2153/#2156/#2195).** The `prSha`-based route was found wrong; the reliable route is an org-wide Buildkite build search by the `trunk-merge/pr-<N>/` construction-branch prefix. #2156 threaded pipeline+build coordinates into `BuildkiteLogReader` (fixing a bare-`bk job log` defect that failed outside a checkout); #2195 built the eviction-specific `find_trunk_merge_eviction_build` search. Note the engine stores build coordinates, not a pre-fetched excerpt — the worker fetches logs itself.
- **Flake-flavored evictions consume budget (v1 limitation).** Because flake handling is deferred, an infra-flavored eviction (Buildkite timeout/broken agent) currently spends a `ci_attempt_budget` slot like a real failure. Tracked as follow-up.
- **Rate limits are undocumented.** The cadence chosen (a few req/min active, 0 idle) is conservative; the client treats 429 like a transport error with `Retry-After` respect (integer-seconds).
- **Single-token blast radius.** One org token in one keychain entry gates all queue merges. Mitigated by the `trunk status` smoke check and the loud auth-failure attention; accepted for a single-operator deployment.
- **Double observation if a product ever moves between queues.** v1 removes the eviction-vs-GitHub-rebounce double-fire class structurally: `merge_mechanism = trunk_queue` disables the GH-native `mergeQueueEntry`/rebounce observation for the product, so at most one queue observer is ever live per repo. The remaining mono-switch prerequisites (multi-org tokens, 429 handling at mono volume) are listed in non-goals.
- **Humans and Boss sharing the queue.** `/trunk merge` comments and the enqueue label keep working. Boss only tracks entries it has intents for; an externally-enqueued Boss-tracked PR merges via the normal GitHub-side observation. An external `/trunk cancel` on a Boss-submitted entry reads as `cancelled` → intent retired + attention, which is the desired behavior.
